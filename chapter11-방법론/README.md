# 소프트웨어 개발 방법론

## Agile
> 소프트웨어 개발 방법에 있어서 아무런 계획이 없는 개발 방법과 계획이 지나치게 많은 개발 방법들 사이에서 타협점을 찾고자 하는 방법론

### 정의
-  기존 계획이 없는 방법론은 전체적인 개발의 흐름을 느리게 함
   - 미래를 예측하기 힘들고 효율적이지 못함
   - 계획에 너무 의존하는 경우 형식적인 절차를 따르는데 많은 시간과 비용이 소요

- 이를 극복하기 위해 Less Document-oriented, 즉 문서를 통한 개발 방법이 아니라 Code-oriented, 실질적인 코딩을 통한 방법론이 등장

- 애자일 개발 방법론과 과거의 방법론의 차이점
  - 미래를 예측하며 개발을 하지 않고, 일정한 주기로 끊임없이 `프로토타입`(Prototype)을 작성
  - 그때 그때 필요한 요구를 더하고 수정하여 하나의 커다란 소프트웨어를 개발해 나가는 Adaptive Style

- 애자일 개발 프로세스란 어느 특정 개발 방법론을 가리키는 말이 아님
  - `애자일`(Agile, 기민하다는 뜻으로 좋은것을 빠르고 낭비없게 만드는 것)은 개발을 가능하게 해 주는 다양한 방법론 전체를 일컫는 말
  - 예전에는 `경량`(Lightweight) 프로세스로 불림
    - 대표적인 방법 : `익스트림 프로그래밍`(XP:eXtreme Programming)

<br>

### 기존 개발 프로세스와의 차이
- 전통적인 개발 프로세스들은 폭포수 모델과 계획 기반 개발을 따르지만 애자일 개발 프로세스는 그와 반대

- 폭포수 모델과 계획 기반 개발 기법들은 일련의 차례와 탄탄한 계획을 기반으로 하여 개발을 진행시킴
  - 이해하기도, 사용하기도 쉬운 바람직한 기법이기도 하나 계획대로 진행되지 않을 수 있음
  - 예 : 납기일 전 철야, 철야에도 불구하고 납기일 지연, 지연에 따른 비난과 스트레스로 개발자 에너지 소진, 결국 납품된 솔루션은 고객의 요구를 충족하지 못함

- 전통적인 개발 프로세스들은 공업에서 사용하는 정형적 프로세스 제어 모델을 따르고 있음
  - 정형적 프로세스 제어 모델은 동일한 입력에 대해서 동일한 결과가 기대 될 경우에 적합
  - 항상 불확실성을 수반하지만 애자일 개발 프로세스는 경험적 프로세스 제어모델로 개발을 관리

- 일반적으로 애자일 개발 프로세스는 자사가 대중에게 직접 제공하는 서비스를 단계적으로 개선하여 나가려는 입장에서 주로 사용
  - 소프트웨어 개발 업무를 수주하여 납품하는 것을 주 업무로 행하는 개발 업체의 업무의 범위에서 그 특성을 한정하여 비교하는 것은 바람직하지 않음
